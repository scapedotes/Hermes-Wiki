## System Prompt Construction

`AIAgent._build_system_prompt()` concatenates `prompt_parts` in a fixed order. Finally, `"\n\n".join(prompt_parts)` returns the complete system prompt. The observed structure is shown in [[prompt-builder-architecture]], assembled in this order:

1.  **SOUL.md** — Agent Identity (`~/.hermes/SOUL.md`, defaults to `DEFAULT_AGENT_IDENTITY` if not found)
2.  **Tool Use Enforcement Guidelines** — Filtered by model family
3.  **Model-Specific Execution Guidelines** — Specific to OpenAI/Google, etc.
4.  **User/Gateway System Message** — If `system_message` is passed to `run_conversation`
5.  **Memory Usage Guidelines** — Instructs the model on how to use memory tools
6.  **MEMORY Snapshot** — `~/.hermes/memories/MEMORY.md` (Frozen)
7.  **USER PROFILE Snapshot** — `~/.hermes/memories/USER.md` (Frozen)
8.  **External Memory Provider Block** — mem0/honcho/holographic, etc., if enabled
9.  **Skills Index** — Generated by scanning `~/.hermes/skills/`
10. **Project Context Files** — `.hermes.md → AGENTS.md → CLAUDE.md → .cursorrules` (first match wins)
11. **Session Metadata** — Timestamp, Model, Provider, Session ID
12. **Platform Hints** — `PLATFORM_HINTS[platform]`
13. **Conversation Context** — Gateway injected source, Home Channel, delivery options

**Caching Mechanism**: The system prompt is built only once per session (`self._cached_system_prompt`) and rebuilt only after context compression, ensuring reuse across conversation turns to maximize LLM prefix cache hit rate.

**Memory Freezing Mode**: The content of MEMORY.md / USER.md at layers 6-7 is a **snapshot at load time**. Even if the model writes new memories during the conversation, they will not be reflected in the current session's system prompt; they will only take effect in the next session. This is a deliberate design choice to protect the prefix cache.

## Execution Guidelines

### General Tool Use Enforcement Guidelines

```text
# Tool-use enforcement
You MUST use your tools to take action — do not describe what you would do
or plan to do without actually doing it.
```

Applicable models: `gpt`, `codex`, `gemini`, `gemma`, `grok`

### Additional Guidelines for OpenAI Models

```xml
<tool_persistence>
- Use tools whenever they improve correctness
- Do not stop early when another tool call would improve the result
- Keep calling tools until: task complete AND verified
</tool_persistence>

<prerequisite_checks>
- Check whether prerequisite discovery steps are needed
- Do not skip prerequisite steps
</prerequisite_checks>

<verification>
- Correctness: does output satisfy every requirement?
- Grounding: are factual claims backed by tool outputs?
- Formatting: does output match requested format?
- Safety: confirm scope before executing side effects
</verification>
```

### Operational Guidelines for Google Models

-   Always use absolute paths
-   Verify before modifying (read_file/search_files)
-   Dependency checks (do not assume libraries are available)
-   Parallel tool calls
-   Non-interactive commands (-y, --yes flags)

## Context File Injection

**Two independent loading paths**:

| File             | Location                  | Search Scope                                |
| :--------------- | :------------------------ | :------------------------------------------ |
| **SOUL.md**      | `~/.hermes/SOUL.md`       | Globally unique path, loaded independently (Agent Identity Slot) |
| **.hermes.md**   | cwd up to git root        | Project-level configuration, Priority 1     |
| **AGENTS.md**    | cwd only                  | Codebase development guidelines, Priority 2 |
| **CLAUDE.md**    | cwd only                  | Anthropic-compatible format, Priority 3     |
| **.cursorrules** | cwd only                  | Cursor-compatible format, Priority 4        |

**Project context files are mutually exclusive** (first match wins) — the first one found is used, subsequent ones are not loaded. SOUL.md does not participate in this competition; it always coexists with project files.

**Ways to control loading**:
-   `TERMINAL_CWD` — In Gateway mode, determines which directory to search for project files. Defaults to `Path.home()`
-   `skip_context_files=True` — Completely skips loading project files (Common for sub-agents)
-   `build_context_files_prompt(skip_soul=True)` — Skips SOUL.md (Used when already loaded as identity slot)

## Skills Index Injection

The skills index is part of the system prompt, assembled into `prompt_parts` by `_build_system_prompt()` calling `build_skills_system_prompt()`, forming the complete system prompt along with identity, memory, context files, etc.

The system prompt is built only once per session (cached in `self._cached_system_prompt`) and rebuilt only after context compression, ensuring reuse across conversation turns to maximize LLM prefix cache hit rate.

## Skills Prompt Two-Layer Caching

Building the skills index text requires scanning all files under `~/.hermes/skills/` and parsing their frontmatter. To avoid redundant file I/O, a two-layer cache is used for acceleration:

| Layer   | Storage                                       | Hit Condition                                      | Invalidation Condition |
| :------ | :-------------------------------------------- | :------------------------------------------------- | :--------------------- |
| Layer 1 | In-memory LRU (`OrderedDict`, max 8 entries) | cache_key match (skills_dir + tools + toolsets + platform) | Process restart        |
| Layer 2 | Disk Snapshot (`.skills_prompt_snapshot.json`) | mtime + size manifest validation successful        | Skill file change      |

If neither layer is hit, a full filesystem scan is performed → writes back to disk snapshot + writes to in-memory cache.

**Note the distinction**: This cache optimizes the I/O speed of "generating index text," which is a different layer from the LLM API layer's prefix cache (reusing computed tokens).
